### Potential Challenges and Mitigations

1. **Subordinate + HAProxy routing:** A subordinate charm shares the machine with the principal. HAProxy needs to know the machine's IP. The `HaproxyRouteRequirer` library auto-detects this via Juju network bindings, but since the subordinate shares the principal's network, this should work naturally. Needs integration testing to confirm.

2. **Library fetching at build time:** The `haproxy-route` library needs to be fetched via `charmcraft fetch-lib` and committed to `charm/lib/`. The `charmcraft pack` build process (which runs in LXD) can access this path. This is a different pattern from the `charmlibs-interfaces-*` PyPI packages already in use.

3. **Server restart on certificate rotation:** When certificates are renewed, the `certificate_available` event fires again. The charm must write new files and restart the server. The existing `restart()` pattern in `mcp_server.py` handles this cleanly.

4. **OAuth resource_server_url coordination:** When behind HAProxy with TLS, the OAuth `resource_server_url` must be the external HTTPS URL, not `http://localhost:8081`. The charm needs to detect whether an external URL is available (from `get_proxied_endpoints()`) and prefer it. If direct TLS is used instead, it should use `https://<external-hostname>:<port>`.

5. **Path prefix and MCP protocol:** MCP clients send requests to `/mcp/` (streamable HTTP). If a path prefix is configured, the full path becomes `/<prefix>/mcp/`. The MCP client needs to be configured with this full URL. The server must strip the prefix before handling, or mount the MCP app under the prefix correctly.

Analysis: MCP Server Charm for Juju

This is a compelling idea. Let me break down what I understand and the key design questions.

The core value proposition

Charm authors want their workloads to expose functionality (tools, prompts, resources) via MCP without having to: 1. Implement MCP protocol handling themselves 2. Set up HTTP transport, SSE/streamable HTTP, auth 3. Deal with ingress configuration

Your charm would act as a sidecar MCP gateway — a generic MCP server that other charms register their capabilities with via Juju integrations.

The big architecture question: request routing

You identified the key question — where do MCP requests actually go? There are two models:

Option A: MCP Server Charm as proxy (hub-and-spoke) - Client → Traefik → MCP Server Charm → executes calls against related charms' workloads - The MCP server charm receives tool calls and forwards them (e.g., via HTTP, exec, or another protocol) to the actual workload - Pro: Single MCP endpoint, centralized auth/logging - Con: The MCP server needs to know how to invoke each workload's functionality — this is essentially reimplementing an RPC layer

Option B: MCP Server Charm as registry, direct calls to workloads - The MCP server handles list_tools, list_prompts, etc. by aggregating what related charms declared via integration data - But actual call_tool requests get proxied to the workload's own HTTP endpoint - Pro: Workloads handle their own logic, MCP charm is thinner - Con: Each workload still needs an HTTP endpoint for tool execution

Option C: MCP Server Charm does everything (the "declarative" model) - Charms declare tools as command-line invocations or HTTP calls in integration data (e.g., "tool backup-db runs pg_dump ...") - The MCP server charm executes those commands/requests itself - Pro: Workloads don't need to implement anything beyond what they already have (CLI tools, REST APIs) - Con: Security surface, complexity of mapping arbitrary commands to MCP tool schemas

My recommendation

Option C is the most interesting and highest-value, because it truly achieves your goal: workloads don't need to "do the implementation" — they just declare what's available. The integration data would look something like:

tools:
  - name: list-databases
    description: "List all PostgreSQL databases"
    input_schema: {}
    handler:
      type: exec  # or "http"
      command: ["psql", "-l", "--csv"]
      # or: url: "http://{{unit_ip}}:8080/api/databases"

The MCP server charm would: 1. Aggregate tool/prompt/resource declarations from all related charms 2. Serve them over streamable HTTP (the modern MCP transport) 3. On call_tool, execute the declared handler (exec into the workload container, or call its HTTP API) 4. Handle ingress via Traefik integration

Potential concerns

  • Security: Executing arbitrary commands from integration data is powerful but needs guardrards. Juju's trust model helps here since integrations are admin-approved.
  • Exec into workloads: Cross-unit/cross-application exec isn't natively supported in Juju. The workload charm would need to expose an API, or the MCP server runs as a sidecar on the same machine/pod.
  • Sidecar vs standalone: If it's a standalone charm, it can only reach workloads via network (HTTP). If it's a sidecar/subordinate, it could exec into containers but only on the same unit.

A practical starting point

A subordinate charm that: 1. Deploys an MCP server (e.g., Python with the mcp SDK) as a sidecar container or workload 2. Reads tool/prompt/resource definitions from integration data provided by the principal charm 3. For exec-type handlers, runs commands in the principal's workload container (via Pebble in k8s charms) 4. For http-type handlers, calls the principal's API 5. Integrates with Traefik for ingress

This feels like a genuinely useful pattern. Shall I start drafting a plan with the repo structure, charm metadata, integration schema, and implementation approach?

i'm mot sure that auth is sufficient. when using an mcp server with something like claude desktop, don't we need to do an oauth flow? not sure if an integration could do that for us or if we would need to handle it in the workload.

Good question. The MCP spec defines an OAuth 2.1 flow for remote servers. When Claude Desktop connects to a remote MCP server over streamable HTTP, it expects:

  1. Discovery at /.well-known/oauth-authorization-server
  2. Dynamic client registration
  3. PKCE authorization code flow
  4. Token endpoint for exchanging codes for access tokens

Our simple Bearer token works for pre-shared secrets (e.g. server-to-server, or a user manually configuring a known token), but won't work for interactive clients like Claude Desktop that expect the full OAuth flow.

Options:

  1. Workload handles it — FastMCP has built-in OAuth support via OAuthServerProvider. The workload would need to implement the authorization server (consent screen, token storage, etc.). This is self-contained but means we're building an auth server.

  2. Integration with an identity provider — Use a Juju relation to an existing OAuth/OIDC provider (e.g. Canonical Identity Platform, Hydra, Keycloak). The MCP server becomes an OAuth resource server that validates tokens issued by the IdP. The IdP handles the authorization flow.

  3. Reverse proxy handles it — When we do Phase 5 (Traefik ingress), OAuth2 Proxy or Traefik's ForwardAuth could handle the flow, and the MCP server just receives pre-authenticated requests.

  4. Hybrid — Keep the Bearer token for internal/simple use, add OAuth as an optional mode for external-facing deployments.

Option 2 is probably most "Juju-native" — there are existing charms for identity providers. Option 1 gives the most control but is more work. Option 3 pushes it to the ingress layer.

Want me to research what FastMCP's OAuthServerProvider looks like and what identity provider charms are available, or would you rather defer this to a later phase and note it on the roadmap?

How it fits together

FastMCP side (workload): - FastMCP supports resource-server-only mode via a TokenVerifier protocol — a single method: verify_token(token) -> AccessToken | None - You pass token_verifier=... (not auth_server_provider) to FastMCP, along with AuthSettings(issuer_url=..., resource_server_url=...) - The SDK automatically serves /.well-known/oauth-protected-resource pointing clients to the external IdP, and enforces Bearer token validation - We just need to implement TokenVerifier to validate JWTs using the IdP's JWKS endpoint (or call its introspection endpoint for opaque tokens)

Juju side (charm): - The oauth relation interface (v0, provided by Hydra) is the standard - Hydra's OAuthRequirer library handles client registration and gives us back: jwks_endpoint, introspection_endpoint, issuer_url, client_id, client_secret_id - The jwt_access_token flag tells us whether to expect JWTs (validate locally via JWKS) or opaque tokens (validate via introspection) - Charmed Kafka already uses this exact pattern

The flow would be: 1. juju integrate mcp-server hydra — the charm gets OIDC endpoints via the oauth relation 2. The charm writes the issuer URL, JWKS endpoint, etc. to the workload config 3.

## 1. DEFINING SUBORDINATE CHARMS IN CHARMCRAFT.YAML

**subordinate: true Field:**
In your `charmcraft.yaml`, set `subordinate: true` to declare that the charm is meant to be deployed as a subordinate to a principal charm rather than standalone.

**Structure:**
```yaml
type: charm
name: my-subordinate-charm
base: ubuntu@24.04
subordinate: true
```

**Machine Charm Bases:**
For machine charms, you must specify the `bases` configuration with build and run specifications. Typical structure:
```yaml
bases:
  - build-on:
      - name: ubuntu
        channel: "20.04"
    run-on:
      - name: ubuntu
        channel: "20.04"
      - name: ubuntu
        channel: "22.04"
```

Real-world example: The `livepatch-machine-charm` shows how to configure bases for Ubuntu 20.04 and 22.04 on amd64 architecture.

**Relation Configuration:**
Your subordinate charm must declare at least one `requires` relation with `scope: container`:

```yaml
requires:
  monitoring:
    interface: monitoring
    scope: container
```

This `scope: container` designation is critical for subordinate charms—it restricts the relation to communicate between a single principal unit and a single subordinate unit, unlike `scope: global` which considers all possible remote units.

## 2. PRINCIPAL CHARM DATA PROVISION ON RELATIONS

**How Subordinates Read Principal Data:**

When a subordinate charm creates a relation with a principal charm:

1. The principal charm writes data to the relation databag via `relation.data[self.app]`
2. The subordinate reads that data using `event.relation.data[event.relation.app]`

**Relation Data Bag API in Ops:**

The ops framework provides structured access through the `.data` attribute as a dictionary:

```python
# Reading remote app data (from principal)
remote_app_data = event.relation.data[event.relation.app]
value = remote_app_data.get('key_name')

# Reading remote unit data
for unit in event.relation.units:
    if unit.app != self.app:
        unit_data = event.relation.data[unit]

# Writing local app data (subordinate publishes to principal)
event.relation.data[self.app]['local_key'] = 'value'
```

**Important Note on Data Format:**
Relation data values are always stored as strings. If you need to pass complex data (JSON, numbers), you must serialize and deserialize accordingly. The relation data bag stores key-value pairs where both keys and values are strings.

**Testing with SubordinateRelation:**
In ops.testing, subordinate relations use `SubordinateRelation` with `remote_app_data`:
```python
relation = testing.SubordinateRelation(
    endpoint='monitoring',
    remote_app_data={'metric_type': 'cpu', 'interval': '60'}
)
```

## 3. BEST PRACTICES FOR SYSTEMD SERVICE MANAGEMENT

**Installation Phase (install event):**
- Use subprocess to call system package managers (apt-get, dnf, etc.)
- Write systemd unit files to `/etc/systemd/system/`
- Always use absolute paths in subprocess calls for security
- Execute processes directly, not through shells

**Startup Phase (start event):**
- Use `subprocess.run(["/usr/bin/systemctl", "enable", "--now", "service-name"])`
- This enables the service AND starts it in one operation
- Wrap in try-except blocks for error handling

**Configuration Phase (config-changed event):**
- Check and reapply configuration regularly (not just during install)
- Can write override files: `/etc/systemd/system/[service-name].service.d/override.conf`
- Reload systemd daemon with `systemctl daemon-reload` if config files change

**Status Reporting:**
- Use `MaintenanceStatus` during operations
- Use `BlockedStatus` when errors occur
- Use `ActiveStatus` when fully operational

**Error Handling:**
- Log return codes and stderr when errors occur
- Put the charm in blocked state to signal deployment issues
- Check service status regularly (not just once during install)

**Core Concepts:**

The relation object has a `.data` attribute that acts as a dictionary containing databags for all participants:

```python
def _on_relation_changed(self, event: RelationChangedEvent) -> None:
    # Local app databag (this charm's application-level data)
    local_app_databag = event.relation.data[self.app]

    # Remote app databag (principal/requirer charm's app data)
    remote_app_databag = event.relation.data[event.relation.app]

    # Local unit databag (this charm's unit-level data)
    local_unit_databag = event.relation.data[self.unit]

    # Remote units (iterate through related units)
    for unit in event.relation.units:
        if unit.app != self.app:
            remote_unit_data = event.relation.data[unit]
```

class MySubordinateCharm(ops.CharmBase):

# MCP Server Charm for Juju

A subordinate machine charm that deploys a [Model Context Protocol](https://modelcontextprotocol.io/) (MCP) server, allowing principal charms to declaratively expose tools, prompts, and resources to MCP clients (LLM agents, Claude Code, etc.) — without implementing MCP themselves.

## How it works

```
                  ┌─────────────────────────────────────────────┐
                  │              Machine (Juju unit)            │
                  │                                             │
  Client ──► Traefik ──► MCP Server (subordinate charm)        │
                  │            │                                │
                  │            │ exec / http                    │
                  │            ▼                                │
                  │       Principal workload                    │
                  │       (e.g. PostgreSQL, app, etc.)          │
                  └─────────────────────────────────────────────┘
```

1. A principal charm (e.g. PostgreSQL) **provides** the `mcp` interface, declaring its tools, prompts, and resources in the relation data bag.
2. The MCP server charm attaches as a **subordinate** (via `scope: container`) and reads those declarations.
3. It runs a Python MCP server (using the [MCP Python SDK](https://github.com/modelcontextprotocol/python-sdk)) as a **systemd service** on the same machine.
4. When an MCP client calls a tool, the server executes the declared handler — either a shell command (`exec`) or a local HTTP call (`http`).
5. Ingress is provided via **Traefik** integration.

## Integration interface

The principal charm provides tool/prompt/resource definitions as JSON in the `mcp` relation's app data bag. Example:

```json
{
  "tools": [
    {
      "name": "list-databases",
      "description": "List all PostgreSQL databases",
      "input_schema": {
        "type": "object",
        "properties": {},
        "required": []
      },
      "handler": {
        "type": "exec",
        "command": ["sudo", "-u", "postgres", "psql", "-l", "--csv"]
      }
    }
  ],
  "prompts": [],
  "resources": []
}
```

### Handler types

- **`exec`** — Run a command on the machine. Arguments are substituted into `{{placeholder}}` positions in the command array. Uses `subprocess.run` with list args (no shell injection).
- **`http`** — Call a local HTTP endpoint. Supports method, URL, and body templates.

See [docs/integration-schema.md](docs/integration-schema.md) for the full schema reference.

# MCP Integration Schema Reference

This document defines the data schema that a principal charm must provide on the `mcp` relation's app data bag.

## Top-level structure

The principal charm sets a JSON string on the relation app data bag under the key `"mcp_definitions"`. The JSON has this structure:

```json
{
  "tools": [...],
  "prompts": [...],
  "resources": [...]
}
```

All three fields are optional. If omitted, they default to empty lists.

## Tools

Each tool declares a function that MCP clients can invoke.

```json
{
  "name": "tool-name",
  "description": "Human-readable description of what this tool does",
  "input_schema": {
    "type": "object",
    "properties": {
      "param1": {"type": "string", "description": "A parameter"},
      "param2": {"type": "integer", "description": "Another parameter"}
    },
    "required": ["param1"]
  },
  "handler": { ... }
}
```

| Field          | Type   | Required | Description |
|----------------|--------|----------|-------------|
| `name`         | string | yes      | Unique tool name (lowercase, hyphens OK) |
| `description`  | string | yes      | What the tool does |
| `input_schema` | object | yes      | JSON Schema for the tool's input parameters |
| `handler`      | object | yes      | How to execute this tool (see Handler types) |

## Prompts

Each prompt declares a reusable prompt template.

```json
{
  "name": "prompt-name",
  "description": "Human-readable description",
  "arguments": [
    {
      "name": "arg1",
      "description": "Argument description",
      "required": true
    }
  ],
  "template": "Analyze the {{arg1}} and provide recommendations."
}
```

| Field         | Type   | Required | Description |
|---------------|--------|----------|-------------|
| `name`        | string | yes      | Unique prompt name |
| `description` | string | yes      | What the prompt is for |
| `arguments`   | list   | no       | List of prompt arguments |
| `template`    | string | yes      | Prompt text with `{{arg}}` placeholders |

## Resources

Each resource declares a readable data source.

```json
{
  "uri": "config://myapp",
  "name": "My App Configuration",
  "description": "Current application configuration",
  "mime_type": "text/plain",
  "handler": { ... }
}
```

| Field         | Type   | Required | Description |
|---------------|--------|----------|-------------|
| `uri`         | string | yes      | Resource URI (unique identifier) |
| `name`        | string | yes      | Human-readable name |
| `description` | string | yes      | What this resource provides |
| `mime_type`   | string | no       | MIME type of content (default: `text/plain`) |
| `handler`     | object | yes      | How to fetch the resource content |

## Handler types

### `exec` — Run a shell command

```json
{
  "type": "exec",
  "command": ["sudo", "-u", "postgres", "psql", "-d", "{{database}}", "-c", "{{query}}"],
  "timeout": 30,
  "user": "root",
  "working_dir": "/tmp",
  "env": {"PGPASSWORD": "secret"}
}
```

| Field         | Type   | Required | Default | Description |
|---------------|--------|----------|---------|-------------|
| `type`        | string | yes      |         | Must be `"exec"` |
| `command`     | list   | yes      |         | Command as argv list. Supports `{{param}}` substitution. |
| `timeout`     | int    | no       | 60      | Max execution time in seconds |
| `user`        | string | no       | root    | Run command as this OS user |
| `working_dir` | string | no       |         | Working directory for the command |
| `env`         | object | no       |         | Extra environment variables |

**Template substitution:** `{{param_name}}` in the command array elements is replaced with the corresponding value from the MCP tool call arguments. Each substituted value becomes a discrete argv element — there is no shell interpolation. Values are validated against `input_schema` before substitution.

### `http` — Call a local HTTP endpoint

```json
{
  "type": "http",
  "url": "http://localhost:8080/api/query",
  "method": "POST",
  "headers": {"Content-Type": "application/json"},
  "body_template": "{\"query\": \"{{query}}\", \"database\": \"{{database}}\"}",
  "timeout": 30
}
```

| Field           | Type   | Required | Default | Description |
|-----------------|--------|----------|---------|-------------|
| `type`          | string | yes      |         | Must be `"http"` |
| `url`           | string | yes      |         | URL to call (typically localhost) |
| `method`        | string | no       | GET     | HTTP method |
| `headers`       | object | no       |         | Extra HTTP headers |
| `body_template` | string | no       |         | Request body with `{{param}}` substitution |
| `timeout`       | int    | no       | 30      | Request timeout in seconds |

## Complete example

Here is a full example of what a PostgreSQL charm might set on the relation:

```json
{

    {
      "name": "list-databases",
      "description": "List all PostgreSQL databases",
      "input_schema": {"type": "object", "properties": {}, "required": []},
      "handler": {
        "type": "exec",
        "command": ["sudo", "-u", "postgres", "psql", "-l", "--csv"],
        "timeout": 10
      }
    },
    {
      "name": "run-query",
      "description": "Run a read-only SQL query against a database",
      "input_schema": {
        "type": "object",
        "properties": {
          "query": {"type": "string", "description": "SQL SELECT query to run"},
          "database": {"type": "string", "description": "Target database name"}
        },
        "required": ["query", "database"]
      },
      "handler": {
        "type": "exec",
        "command": ["sudo", "-u", "postgres", "psql", "-d", "{{database}}", "-c", "{{query}}", "--csv"],
        "timeout": 30
      }
    }
  ],
  "prompts": [
    {
      "name": "analyze-database",
      "description": "Analyze database health and performance",
      "arguments": [
        {"name": "database", "description": "Database to analyze", "required": true}
      ],
      "template": "Please analyze the health and performance of the '{{database}}' PostgreSQL database. List all tables with their sizes, check for bloat, and identify any potential issues."
    }
  ],
  "resources": [
    {
      "uri": "config://postgresql/main",
      "name": "PostgreSQL Configuration",
      "description": "Current postgresql.conf contents",
      "mime_type": "text/plain",
      "handler": {
        "type": "exec",
        "command": ["cat", "/etc/postgresql/14/main/postgresql.conf"]
      }
    }
  ]
}
```

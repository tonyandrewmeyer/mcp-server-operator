## Research Findings: FastMCP OAuth 2.1 Support

### SDK Version Examined

MCP Python SDK v1.26.0, installed at `/home/ubuntu/mcp-charm/workload/.venv/lib/python3.12/site-packages/mcp/`.

---

### 1. The `OAuthAuthorizationServerProvider` Interface

Defined in `/home/ubuntu/mcp-charm/workload/.venv/lib/python3.12/site-packages/mcp/server/auth/provider.py`, it is a `Protocol` (structural typing) that is generic over three type variables (`AuthorizationCodeT`, `RefreshTokenT`, `AccessTokenT`). It requires these async methods:

| Method | Purpose |
|---|---|
| `get_client(client_id) -> OAuthClientInformationFull | None` | Look up a registered OAuth client |
| `register_client(client_info)` | Store a dynamically registered client (RFC 7591) |
| `authorize(client, params) -> str` | Handle the `/authorize` endpoint; returns a redirect URL. The docstring explicitly describes a double-OAuth pattern where the MCP server redirects to a 3rd-party IdP |
| `load_authorization_code(client, code) -> AuthorizationCode | None` | Retrieve a stored auth code |
| `exchange_authorization_code(client, code) -> OAuthToken` | Exchange auth code for access + refresh tokens |
| `load_refresh_token(client, token) -> RefreshToken | None` | Load a stored refresh token |
| `exchange_refresh_token(client, refresh_token, scopes) -> OAuthToken` | Rotate refresh token for new tokens |
| `load_access_token(token) -> AccessToken | None` | Validate/load an access token |
| `revoke_token(token)` | Revoke an access or refresh token |

When you provide an `OAuthAuthorizationServerProvider`, the SDK creates all the OAuth endpoints (`/authorize`, `/token`, `/register`, `/revoke`, `/.well-known/oauth-authorization-server`) and the MCP server acts as **both** the authorization server and the resource server.

---

### 2. Resource-Server-Only Mode (External IdP Integration)

The SDK supports **AS/RS separation** — the MCP server can act purely as a resource server while delegating authorization to an external IdP. This is the key mechanism:

**`TokenVerifier` Protocol** (also in `provider.py`):

```python
class TokenVerifier(Protocol):
    async def verify_token(self, token: str) -> AccessToken | None:
        """Verify a bearer token and return access info if valid."""
```

In the `FastMCP` constructor (`server/fastmcp/server.py`, line 147), there are two mutually exclusive parameters:

- **`auth_server_provider`** — Use this when the MCP server IS the authorization server (creates all OAuth endpoints)
- **`token_verifier`** — Use this when the MCP server is ONLY a resource server (validates tokens issued elsewhere)

You cannot pass both. The validation logic (line 218-224):
- If `auth` settings are provided, you must provide exactly one of the two
- If `auth_server_provider` is given without `token_verifier`, a `ProviderTokenVerifier` wrapper is automatically created

**To integrate with an external IdP (Keycloak, Hydra, etc.):**

1. Implement the `TokenVerifier` protocol with a single `verify_token` method. This would typically do one of:
   - **JWT validation**: Verify the token signature using the IdP's JWKS endpoint, check claims (audience, expiry, scopes)
   - **Token introspection**: Call the IdP's introspection endpoint (RFC 7662)

2. Pass it to FastMCP along with `AuthSettings`:

### 3. HTTP-Level Requirements per the MCP Spec (2025-06-18)

The MCP authorization spec defines these roles and endpoints:

**MCP Server as Resource Server (MUST):**
- Serve `/.well-known/oauth-protected-resource` (RFC 9728) — tells clients which authorization server(s) to use
- Return `WWW-Authenticate` header on 401 responses pointing to the resource metadata URL
- Validate Bearer tokens in `Authorization` headers
- Validate token audience (the token must have been issued specifically for this MCP server)

**Authorization Server (separate or co-located, MUST):**
- Serve `/.well-known/oauth-authorization-server` (RFC 8414) — metadata including endpoints
- `/authorize` — authorization endpoint
- `/token` — token endpoint
- `/register` — dynamic client registration (SHOULD, RFC 7591)
- `/revoke` — token revocation (optional)

**When using `auth_server_provider`**, the SDK creates all of these on the MCP server itself. **When using `token_verifier`**, it only creates the protected resource metadata endpoint and bearer token validation — the AS endpoints must exist on the external IdP.

The SDK automatically handles:
- Building and serving the `ProtectedResourceMetadata` at `/.well-known/oauth-protected-resource{/path}` (via `create_protected_resource_routes`)
- Including `resource_metadata` in `WWW-Authenticate` headers on 401 responses (via `RequireAuthMiddleware`)
- Bearer token extraction and validation (via `BearerAuthBackend` middleware)
- Scope enforcement (via `RequireAuthMiddleware`)

---

### 4. Can the MCP Server Act Purely as a Resource Server?

**Yes, absolutely.** Both the MCP spec and the SDK explicitly support this.

From the MCP spec (2025-06-18): *"The implementation details of the authorization server are beyond the scope of this specification. It may be hosted with the resource server or a separate entity."*

From the SDK source (provider.py, line 288-294): The `ProviderTokenVerifier` class docstring says: *"This is provided for backwards compatibility with existing auth_server_provider configurations. For new implementations using AS/RS separation, consider using the TokenVerifier protocol with a dedicated implementation like IntrospectionTokenVerifier."*

The recommended approach for RS-only mode:
1. Implement `TokenVerifier` to validate tokens from your external IdP
2. Pass `token_verifier=...` (not `auth_server_provider`) to `FastMCP`
3. Set `AuthSettings.issuer_url` to your IdP's URL and `resource_server_url` to the MCP server's URL
4. The SDK will serve protected resource metadata pointing clients to the external IdP and enforce bearer token validation on all MCP endpoints

## 3. Existing charm library for OAuth integration

The canonical library is published by the **Hydra charm** at:

`https://charmhub.io/hydra/libraries/oauth/source-code` (lib v0.11)

It provides two classes:

- **`OAuthRequirer`** -- For charms that need to act as an OAuth2 client (or resource server). Provides methods: `update_client_config()`, `is_client_created()`, `get_provider_info()`, `get_client_secret()`.

## 4. What data is exchanged via the `oauth` relation

### Requirer provides (app databag):
- `redirect_uri` -- HTTPS callback URL
- `scope` -- Space-separated requested scopes
- `grant_types` -- Array (e.g. `["authorization_code"]`, `["client_credentials"]`)
- `token_endpoint_auth_method` -- e.g. `"client_secret_basic"`
- `audience` -- Array of audiences for issued tokens (optional)

### Provider provides (app databag):
- `issuer_url` -- OIDC issuer URL
- `authorization_endpoint` -- URL for auth flows
- `token_endpoint` -- Token exchange URL
- `introspection_endpoint` -- Token introspection URL
- `userinfo_endpoint` -- User info URL
- **`jwks_endpoint`** -- JWKS public keys URL (for local JWT validation)
- `scope` -- Supported scopes
- `client_id` -- Registered client identifier
- `client_secret_id` -- **Juju Secret ID** referencing the client secret (requires Juju 3+)
- `groups` -- Optional group membership claim
- `ca_chain` -- Optional CA certificate array
- **`jwt_access_token`** -- Optional boolean indicating whether access tokens are JWTs (as opposed to Hydra's default opaque tokens)

### Client registration flow:
1. Requirer charm puts its `ClientConfig` in the relation databag.
2. Provider (Hydra) receives a `client_created` event, registers the OAuth client.
3. Provider stores the client secret in a **Juju Secret**, puts the `client_id` and `client_secret_id` in the databag.
4. Requirer receives `oauth_info_changed` event, retrieves all provider endpoints and credentials via `get_provider_info()`.

### Implications for a resource server charm:
A charm acting as a resource server (validating tokens, not redirecting users) would use the `oauth` relation to obtain:
- The **`jwks_endpoint`** for local JWT signature validation
- The **`introspection_endpoint`** for opaque token validation
- The **`issuer_url`** for `iss` claim validation
- The `client_id` / `client_secret_id` if it needs to call the introspection endpoint (which requires client credentials)

The `jwt_access_token` field tells the resource server whether to expect JWTs (validate locally via JWKS) or opaque tokens (validate via introspection). Hydra defaults to opaque tokens but can be configured per-client to issue JWTs.

The Charmed Kafka integration demonstrates this pattern: `juju integrate kafka hydra` sets up the `oauth` relation, and Kafka then validates bearer tokens from clients using the endpoints provided by Hydra.
